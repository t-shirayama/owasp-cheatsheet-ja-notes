## 日本語訳

### 概要

エラーハンドリングは、アプリケーション全体のセキュリティの一部です。攻撃者は偵察段階で、アプリケーションサーバー、フレームワーク、ライブラリ、バージョン、内部パス、SQL エラーなどの技術情報を集めようとします。未処理例外や詳細すぎるエラー応答は、この偵察を助けます。

エラー処理の問題は、技術スタックの特定だけでなく、入力値がどの処理に渡されるか、どこに注入点があり得るかを示す手がかりにもなります。したがって、ユーザー向け応答とサーバー側調査情報は分離する必要があります。

### 目的

予期しないエラーが発生した場合、アプリケーションはユーザーへ汎用的な応答を返し、詳細なエラー情報はサーバー側にログとして記録します。ユーザーへ返すレスポンスには、実装詳細、スタックトレース、内部パス、SQL 文、ライブラリ名、バージョン、設定値などを含めません。

API ベースのアプリケーションでは、すべての失敗モードを可能な限り網羅し、処理できない要求にだけ 5xx 系の応答を使います。クライアント起因の不正な要求には、状況に応じて 4xx 系の応答を返します。HTTP API では、必要に応じて Problem Details などの標準的なエラー表現を検討します。

### グローバルエラーハンドラ

アプリケーションには、予期しない例外を捕捉するグローバルエラーハンドラを設定します。フレームワークによっては、ランタイム設定、デプロイ記述子、ミドルウェア、例外ハンドラクラス、専用コントローラなどで実装します。

グローバルエラーハンドラの責務は、例外の詳細をサーバー側へ記録し、クライアントへは安全で一貫した応答を返すことです。正常な業務エラー、入力検証エラー、認証・認可エラー、システム障害を同じ扱いにせず、適切な HTTP ステータス、エラーコード、メッセージを設計します。

### ユーザーへ返す情報

ユーザーへ返すエラーは、解決に必要な最小限の情報にします。たとえば、入力が不正であればどの入力が修正対象かを説明できますが、内部例外、スタックトレース、SQL エラー、ファイルパス、ホスト名、パッケージ名、バージョンは返しません。

認証や認可に関するエラーでは、攻撃者がアカウント存在確認や権限構造の推測に使える情報を返さないようにします。リソースが存在しない場合と権限がない場合を同じように扱うべきケースもあります。

### サーバー側に記録する情報

調査に必要な詳細は、ユーザーではなくサーバー側のログへ記録します。ログには、例外種別、発生箇所、相関 ID、リクエスト ID、ユーザーまたは主体、対象リソース、入力の分類、HTTP メソッド、パス、ステータス、発生時刻を含めます。

ただし、ログにもパスワード、アクセストークン、秘密鍵、セッション ID、決済情報、過度な個人情報を記録してはいけません。ログ記録自体の方針は [ログ記録チートシート](../translations/logging.md) に従います。

### API 応答の一貫性

API では、成功応答と失敗応答の形式を整理します。エラー応答には、クライアントが処理できる安定したエラーコード、ユーザー向けメッセージ、相関 ID などを含めることを検討します。内部例外メッセージをそのまま `message` に詰めて返してはいけません。

相関 ID は、ユーザーがサポートへ問い合わせる際や、運用チームがサーバー側ログを追跡する際に有用です。相関 ID 自体に秘密情報を含めないようにします。

### フレームワーク別の実装

Java Web Application、Spring MVC / Spring Boot、ASP.NET Core、ASP.NET Web API など、多くのフレームワークはグローバルエラー処理の仕組みを提供します。重要なのは、どの技術スタックでも「例外詳細はログへ、ユーザーには汎用応答」という原則を一貫して適用することです。

既定の開発者向けエラーページ、デバッグモード、詳細例外表示は、本番環境で無効化します。本番設定と開発設定が混ざらないよう、環境ごとの設定を管理します。

### HTTP エラー

HTTP ステータスコードは意味に沿って使います。クライアントが修正すべき問題には 4xx、サーバー側で処理できない問題には 5xx を使います。ただし、ステータスコードや本文が過度な内部情報を示さないようにします。

エラー時のレスポンス本文、ヘッダー、キャッシュ制御、CORS、認証・認可エラーの扱いも確認します。エラー応答がブラウザや中間プロキシに不適切にキャッシュされないようにします。
