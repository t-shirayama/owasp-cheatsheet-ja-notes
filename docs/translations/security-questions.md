## 日本語訳

秘密の質問は、NIST SP 800-63 の観点では受け入れ可能な認証要素とはみなされない。アカウント回復は認証の代替経路であり、通常の認証より弱くしてはならない。このチートシートは、安全なソフトウェアで秘密の質問を推奨するものではなく、レガシー用途で残さざるを得ない場合のリスク低減策を示す。

秘密の質問に求められる性質は、記憶できること、時間が経っても変わらないこと、利用者が答えられること、攻撃者が取得しにくいこと、利用者にとって明確であることの5つである。生年月日、好きな映画、好きなスポーツチーム、最初の車、ニックネームのような質問は、公開情報から推測されやすい、変化しやすい、適用できる利用者が限られる、候補が少ないなどの問題がある。

利用者定義の質問は、利用者が強くユニークな質問を作れる可能性がある一方で、弱い質問やパスワードのヒントを設定する危険があるため、原則として許可しない。システム定義の質問は、既に保有している個人情報に基づけるが、氏名、住所、生年月日のような情報は攻撃者が入手しやすく、強い認証にはならない。アプリケーションの文脈によって、同じ質問でも強度は変わる。

秘密の質問は、パスワードと同じ「知っているもの」であり、パスワードとの組み合わせでも MFA にはならない。単独の認証手段として使ってはならない。より強い MFA が使えないレガシー環境で、追加レイヤーとして使う場合に限定する。

忘れたパスワードや失った MFA トークンの回復フローで秘密の質問を使う場合、先に利用者のメール所有を確認してから質問を表示する。メールアドレス入力時は一般的なメッセージを返し、ランダムで単回使用のリンクを送付し、そのリンクをクリックした後に質問を提示する。これにより、回復フローがアカウント列挙に使われることを避ける。

秘密の質問の回答は、個人情報を含み、他サイトで再利用される可能性があるため、パスワードと同じように安全なハッシュで保存する。回答の比較は利用者負荷を下げるため大文字小文字を区別しないことがあるが、その場合は保存前と比較前の正規化を一貫させる。入力形式が曖昧な質問には形式の案内を出す。

秘密の質問や回答を更新する操作は機微な操作として扱う。利用者にパスワード再入力、できれば MFA による再認証を求め、攻撃者が一時的なセッションを得ただけで回復経路を差し替えられないようにする。複数の質問を使う場合は、攻撃者が質問を順番に試せるようにしてはならない。単一質問を質問群から選ぶ方式では、正解するまで質問を変更しない。
