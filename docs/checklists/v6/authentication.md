## 開発チェックリスト

### V6.2 Password Security

- [ ] 生成する: 外部に露出または推測され得る User ID は、連番ではなく予測困難な値にする。
- [ ] 禁止する: バックエンド、ミドルウェア、データベースなどの機密アカウントを公開フロントエンド UI からログイン可能にすること。
- [ ] 強制する: MFA 有効時でも 8 文字未満のパスワードを弱いものとして拒否する。
- [ ] 強制する: MFA がない場合は 15 文字未満のパスワードを弱いものとして拒否する。
- [ ] 許可する: パスフレーズを使えるよう、少なくとも 64 文字の最大長を設定する。
- [ ] 禁止する: パスワードを黙って切り詰めること。
- [ ] 許可する: Unicode と空白を含む文字をパスワードに使用できるようにする。
- [ ] 禁止する: 大文字、小文字、数字、特殊文字などの固定的な構成ルールだけに依存すること。
- [ ] 実装する: 一般的なパスワードと漏えい済みパスワードをブロックする。
- [ ] 実装する: パスワード強度メーターを提供し、ユーザーの強いパスワード選択を支援する。
- [ ] 実装する: パスワード保存を Password Storage Cheat Sheet に従い、平文保存、可逆暗号保存、高速ハッシュだけの保存を禁止する。
- [ ] 実装する: パスワード比較には安全な比較関数を使い、独自実装では最大入力長、明示的な型設定、定数時間比較を満たす。
- [ ] 実装する: パスワード変更時に、有効な認証済みセッションと現在パスワードを検証する。
- [ ] 強制する: ログインページと認証後ページを TLS または同等に強いトランスポートだけで提供する。
- [ ] テストする: 長すぎるパスワード、Unicode、空白、漏えい済みパスワード、現在パスワードなしの変更、平文 HTTP ログインが期待どおり拒否または処理される。

### V6.3 General Authentication Security

- [ ] 実装する: パスワードリカバリは安全なワンタイムトークン、有効期限、アカウント列挙防止、再利用防止を満たす。
- [ ] 要求する: パスワード変更、メール変更、支払い情報変更、信頼済みデバイス追加などの機密操作で再認証を行う。
- [ ] 実装する: アカウントリカバリ、パスワードリセット、不審なログインパターン、IP 変化、デバイス登録などのリスクイベント後に再認証を要求する。
- [ ] ローテーションする: 再認証後に必要に応じてセッションを無効化し、トークンを再発行する。
- [ ] 返却する: ログイン、パスワードリカバリ、アカウント作成で、アカウント存在やロック状態を漏らさない汎用エラーメッセージを返す。
- [ ] 統一する: 認証失敗時の HTTP ステータスコード、エラー URL、処理時間がユーザー列挙の差分にならないようにする。
- [ ] 実装する: ブルートフォース、クレデンシャルスタッフィング、パスワードスプレーに対してスロットリングを行う。
- [ ] 実装する: アカウントロックアウトの失敗カウンターを送信元 IP ではなくアカウントに関連付ける。
- [ ] 設計する: ロックアウトしきい値、観測期間、ロックアウト期間、指数バックオフ、リカバリ手段を明確にする。
- [ ] 防止する: ロックアウト機能が他ユーザーをロックするサービス拒否に使われないようにする。
- [ ] 適用する: CAPTCHA は多層防御として扱い、一定回数の失敗後など攻撃コストを上げる場面で使う。
- [ ] 禁止する: セキュリティ質問や記憶語を MFA とみなすこと。
- [ ] 記録する: すべての認証失敗、パスワード失敗、アカウントロックアウトを監査ログへ記録しレビューする。
- [ ] 禁止する: 認証ログにパスワード、リカバリトークン、セッション ID、MFA コードを平文で記録すること。
- [ ] テストする: アカウント列挙、応答時間差、スロットリング回避、ロックアウト DoS、リカバリトークン再利用、ログ漏えいを確認する。

### V6.5 General Multi-factor authentication requirements

- [ ] 実装する: 可能な限り MFA を提供し、高リスク操作または高リスクイベントで step-up MFA を要求する。
- [ ] 実装する: 支払い、送金、重要設定変更などでは、操作内容と追加認証を結び付けるトランザクション認証を検討する。
- [ ] 確認する: MFA 登録、変更、回復、無効化の手順が、攻撃者の迂回経路になっていない。
- [ ] テストする: MFA なしで高リスク操作が実行できないこと、MFA 回復手順でアカウント奪取できないことを確認する。

### V6.7 Cryptographic authentication mechanism

- [ ] 検討する: 企業内や高特権用途など、利用端末を制御できる場合は TLS クライアント認証を追加保護として評価する。
- [ ] 実装する: TLS クライアント認証を使う場合、ユーザー専用証明書を発行し、証明書の subject などからユーザーを検証する。
- [ ] 確認する: TLS 復号を行う企業プロキシが証明書認証を壊さないよう、ネットワーク構成を確認する。
- [ ] 実装する: FIDO、WebAuthn、Passkey など公開鍵ベースの認証を使う場合、フィッシング耐性、デバイス登録、回復手順を設計する。
- [ ] テストする: 無効証明書、期限切れ証明書、別ユーザー証明書、不正 origin の WebAuthn 認証が拒否される。

### V6.8 Authentication with an Identity Provider

- [ ] 禁止する: サードパーティアプリケーションにユーザー名とパスワードの組み合わせを保存させること。
- [ ] 使い分ける: 認証と SSO には OIDC、API 認可には OAuth を使い、ID Token とアクセストークンを混同しない。
- [ ] 検証する: OIDC の ID Token について issuer、audience、署名、有効期限を relying party 側で確認する。
- [ ] 使用する: OIDC では保守されたライブラリ、provider discovery、JWKS エンドポイントを優先する。
- [ ] 禁止する: 新規システムで OpenID 2.0 を実装すること。
- [ ] 確認する: SAML 2.0 を使う場合、SAML Security Cheat Sheet の署名、XML、リダイレクト、IdP/SP 起点フローの制御を別途確認する。
- [ ] 実装する: パスワードマネージャーを妨げない標準 HTML フォーム、適切な `type` 属性、貼り付け許可、Tab 移動を提供する。
- [ ] 実装する: 登録メールアドレス変更で、現在セッション検証、本人確認、pending change、現在アドレス通知、新アドレス確認を行う。
- [ ] 要求する: MFA がないユーザーのメール変更では現在パスワードと、現在アドレスおよび新アドレスの確認を要求する。
- [ ] 設計する: リスクベース認証で監視する IP、位置情報、デバイス、時刻、行動などの信号、更新頻度、欠損時の扱い、リスク階層、アクションを定義する。
- [ ] 適用する: リスク階層に応じて、許可、CAPTCHA、step-up MFA、ブロック、セッション失効を Web、モバイル、API クライアントで一貫して適用する。
- [ ] テストする: issuer 不一致、audience 不一致、署名不正、期限切れ ID Token、メール変更フローの片側確認漏れ、リスク上昇時の step-up 不備を確認する。
