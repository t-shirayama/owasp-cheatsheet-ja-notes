## 開発チェックリスト

- [ ] 高リスク操作を定義する。
- [ ] 各高リスク操作で、利用者に確認させる重要トランザクションデータを定義する。
- [ ] What You See Is What You Sign の原則に従い、宛先、金額、対象アカウント、権限変更内容などを承認画面または外部認可コンポーネントに表示する。
- [ ] 認可トークン変更時に、現在の認可トークンで変更を承認させる。
- [ ] 認可方式変更時に、現在の認可方式で変更を承認させる。
- [ ] クライアントパラメータで認可方式を弱い方式へダウングレードできないよう、サーバ側で方式を強制する。
- [ ] 認証操作とトランザクション認可操作で、利用者に異なる文脈、文言、操作を提示する。
- [ ] 認証 OTP をトランザクション認可へ流用できないようにする。
- [ ] 各トランザクションに一意の認可資格情報、チャレンジ、タイムスタンプ、シーケンス番号、ランダム値を使う。
- [ ] 認可資格情報に短い有効期限を設定し、チャレンジ生成から承認完了までの時間窓を制限する。
- [ ] 認可資格情報の提出回数を制限し、一定回数失敗したら認可プロセス全体を再開始する。
- [ ] 重要なトランザクション検証データをサーバ側で生成、保存、検証する。
- [ ] クライアントから送られた取引データを承認済みデータとして信頼しない。
- [ ] トランザクション状態遷移をサーバ側で管理し、ステップのスキップ、順序変更、再実行を禁止する。
- [ ] 承認後に取引データが変更された場合は、既存 OTP、チャレンジ、署名要求を失効させる。
- [ ] 承認後の取引データ変更試行を攻撃としてログ記録、監視、調査する。
- [ ] トランザクションデータの機密性と完全性をすべてのクライアントサーバ通信で保護する。
- [ ] 実行直前に、対象トランザクションが正しい利用者と正しいデータで承認済みであることを検証する。
- [ ] TOCTOU を防ぐため、承認対象データと実行対象データを同じサーバ側レコードまたは署名済みデータに結び付ける。
- [ ] 承認、失敗、再試行、データ変更、実行、失効を監査ログへ記録する。
- [ ] 署名鍵や端末ペアリングを使う場合は、鍵の発行、保護、置換、盗難時失効の手順を用意する。
- [ ] テストで、認可方式ダウングレード、認証 OTP 流用、ステップスキップ、承認後改ざん、リプレイ、期限切れ資格情報、総当たりを検証する。
